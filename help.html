APItiser Help

Scan → Generate → Validate → Download

Quick Start

  • Open a GitHub or GitLab repository page in the active tab.
  • Open APItiser from the toolbar icon (opens as a side panel).
  • In Settings, add your LLM provider API key (OpenAI, Claude, or Gemini).
  • Click Scan Repo to detect API endpoints.
  • Review detected endpoints, select/deselect as needed, then click Generate Tests.
  • After generation completes, click Download Tests to save the ZIP artifact.

Settings Reference

Provider & Model

  • Provider — Choose OpenAI, Claude, or Gemini.
  • Model — Select from available models for the chosen provider.
  • API Key — Paste the provider API key. Keys are stored locally and never sent to any APItiser server.
  • Auto-fallback — When enabled, if the primary provider fails, APItiser automatically tries the next configured provider.

Repo Access

  • GitHub Token — Optional for public repos. Required for private repositories or to increase rate limits.
  • GitLab Token — Required for private repositories on GitLab.
  • GitLab Base URL — For self-hosted GitLab instances (defaults to https://gitlab.com).
  • Validate Access — Checks token scopes, repository visibility, and host reachability before scanning.

Test Configuration

  • Framework — Jest, Mocha + Chai, or Pytest. Determines the generated test file format.
  • Batch Size — Number of endpoints sent per LLM request (1–20, default 6).
  • Timeout — Maximum seconds per batch before the request is aborted.
  • Base URL — Injected into generated test files (e.g. http://localhost:3000). Also used for live validation.
  • Test Folders — Comma-separated directories where existing tests live. APItiser checks these to detect already-covered endpoints.
  • Skip Existing Tests — When on, endpoints that already have tests are excluded from generation.
  • Categories — Toggle which test types to generate: positive, negative, edge, and security.
  • Custom Prompt Instructions — Free-text instructions appended to the LLM prompt (e.g. "Always test pagination with limit=0").

Live Validation

  • Validate Generated Tests — When enabled, APItiser executes each generated test against the Base URL before packaging.
  • Auto-Repair — When enabled, tests that fail live validation are sent back to the LLM for automatic repair.
  • Repair Rounds — Maximum number of validation → repair cycles (0–5, default 2).
  • Runtime Auth Mode — Choose how authentication is supplied during live validation: None, Bearer token, API key, Cookie session, or OAuth2.
  • Runtime Credentials — API Token, API Key, Session Cookie, and CSRF Token values used exclusively during live validation.

Validation Setup Flow

For APIs that require login or token exchange before validation, configure setup steps that run before the test suite. Each step can:

  • Send a request to a login or token endpoint.
  • Extract tokens from the JSON response body, response headers, or cookies.
  • Pass extracted values to subsequent validation requests as bearer tokens, API keys, CSRF tokens, or session cookies.

OpenAPI Fallback

  • Paste or import an OpenAPI spec (JSON or YAML) for endpoints that cannot be detected from source code.
  • The fallback spec is merged into scan results alongside code-detected routes.

Settings Backup

  • Export Settings — Download a JSON file with non-sensitive settings. API keys, tokens, runtime credentials, and setup flows are excluded.
  • Import Settings — Upload a previously exported settings file to restore configuration.

Endpoint Selection

After scanning, you can select or deselect individual endpoints for generation. Use the method filter to narrow the list by HTTP method. The Select All and Clear All buttons update the selection in bulk.

Readiness Assessment

After generation, each artifact receives a readiness label:

  • Production Candidate — All tests passed live validation, have strong assertions, and no heuristic-only evidence.
  • Validated — All tests passed live validation, but some rely on heuristic evidence or lack schema assertions.
  • Review Required — Live validation did not run or some tests failed. Manual review recommended.
  • Scaffold — No tests were generated.

Test Preview

After generation completes, click Preview Tests to inspect all generated test cases, their categories, expected status codes, and assertions before downloading.

Postman Export

Click Export Postman to download a Postman collection JSON file based on the generated tests. Useful for manual API exploration alongside automated tests.

Context Behavior

APItiser isolates state per browser tab and page URL. Switching tabs shows that tab's own scan results, generated tests, progress, and history. Each context maintains independent job history and artifacts.

Auto-Resume

If Chrome terminates the service worker during a long-running scan or generation, APItiser automatically resumes from the last checkpoint when the worker restarts. No data is lost.

Clear Action

Clear removes the current page context only: active job, job history, generated tests, and artifacts for that page. Other page contexts and global settings are not affected.

Supported Frameworks

  • Express, Fastify, NestJS, Koa, Hono, Next.js API routes
  • FastAPI, Flask (Python)
  • Spring Boot, Gin (Java/Go)
  • Any API described via an OpenAPI 3.x spec (JSON or YAML)

Troubleshooting

  • If scan fails, use Validate Access in Settings to check token scopes and repository visibility.
  • If generation fails, verify the provider API key and model selection. Check that the key has sufficient quota.
  • If live validation fails, confirm the Base URL is reachable and that runtime credentials are correct.
  • For rate limit issues with GitHub/GitLab, add a personal access token in Settings to increase limits.
  • For repos with unusual metaprogramming patterns, use the OpenAPI Fallback to supply endpoint definitions manually.
  • Reload the extension from chrome://extensions after updating the extension files.
